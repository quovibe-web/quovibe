# ADR-014: Introduction of custom `vf_exchange_rate` table

**Status:** Accepted (architecture) — Remediation complete (2026-04-13)

> **Note (2026-04-15, ADR-015):** this ADR references `extensions.ts`, `applyExtensions()`, and `preserveCustomTables()` which have since been deleted. Under ADR-015 (DB Bootstrap & Portfolio Lifecycle) `vf_exchange_rate`'s DDL moved into `packages/api/src/db/bootstrap.sql` (§3) and is applied via `applyBootstrap(db)` on every `openDatabase()`. The `INSERT OR IGNORE` preservation path during re-import is obsolete: each import now creates a **new** portfolio `.db`, so there is nothing to preserve across. The conceptual decisions in this ADR (dedicated table, `vf_` prefix, TEXT rate column, etc.) still hold — only the implementation entry points changed.

## 1. Context & Problem Statement

Quovibe consumes a SQLite database generated by `ppxml2db`, which converts a Portfolio Performance XML export into 24 relational tables. This database is treated as the imported dataset — a complete snapshot of the user's portfolio as of the last XML export.

The problem: quovibe needs historical exchange rate lookups for multi-currency performance calculations, but the native ppxml2db schema does not provide a clean, general-purpose FX rate table. PP stores exchange rate data in two places, neither of which is suitable:

### (a) Transaction-level FX: `xact_unit.exchangeRate`

Each transaction unit stores the exchange rate that was applied to *that specific transaction* (`packages/api/vendor/xact_unit.sql:8-13`):

```sql
exchangeRate VARCHAR(16)  -- arbitrary-precision, per-transaction
```

This is per-transaction, not reusable. You cannot ask "what was EUR/USD on 2024-03-15?" — only "what rate was used in transaction XYZ's FOREX unit." Performance calculations need dense date-range lookups, not per-transaction spot values.

### (b) Exchange rate securities: `security` + `price`

PP can store exchange rates as special "securities" with `targetCurrency` set (`packages/api/vendor/security.sql:8`), with historical rates in the `price` table. However:

- **Not guaranteed to exist** — these are user-configured in PP; a minimal export may not include them.
- **Query ergonomics are poor** — answering "EUR→USD on date X" requires finding the right security (`WHERE currency = 'USD' AND targetCurrency = 'EUR'`), then joining to `price`, dealing with direction/convention ambiguity.
- **No triangulation** — if the user has EUR→USD and EUR→GBP but not USD→GBP, there's no built-in cross-rate.

### Overwrite-survival hypothesis: CONFIRMED

The critical driver — visible in the code at `packages/api/src/services/import.service.ts`:

1. `runImport()` creates a fresh temporary database from scratch
2. `ppxml2db.py` populates it entirely from the XML file
3. `applyExtensions(newDb)` creates an empty `vf_exchange_rate` via `CREATE TABLE IF NOT EXISTS`
4. `reloadApp()` performs an atomic file swap (`index.ts`): the temp DB replaces the live DB

If quovibe had written live-fetched FX rates into the native `price` table, every re-import would obliterate them. The developer created `vf_exchange_rate` as a sidecar table to decouple live FX data from the PP import lifecycle.

The naming convention confirms intent — the `vf_` prefix (seen in the `extensions.ts` comment: *"Tabelle proprie di quovibe (non ppxml2db)"*) explicitly marks the table as quovibe-owned, outside the ppxml2db namespace.

## 2. Decision Drivers

| # | Driver | Code Evidence |
|---|--------|---------------|
| **D1** | **Separation of concerns: PP-imported vs quovibe-owned data** | `extensions.ts` — Comment: *"Tabelle proprie di quovibe (non ppxml2db)"*. Table defined in its own file, created by `applyExtensions()`, not by ppxml2db. `vf_` prefix reserved for quovibe-only tables. |
| **D2** | **Clean lookup semantics** | `fx.service.ts` — A single prepared statement: `SELECT rate FROM vf_exchange_rate WHERE from_currency = ? AND to_currency = ? AND date <= ? ORDER BY date DESC LIMIT 1`. No security JOINs, no direction ambiguity. Composite PK `(date, from_currency, to_currency)` is purpose-built for this access pattern. |
| **D3** | **EUR-based cross-rate triangulation** | `fx.service.ts` — Three-tier lookup: direct pair → inverse → EUR pivot. Cross-rate formula: `USD→GBP = (EUR→GBP) / (EUR→USD)`. ECB only publishes EUR-based rates; the triangulation layer supports any pair without requiring the user to pre-configure them in PP. |
| **D4** | **Forward-fill for non-trading days** | `fx.service.ts` — Query includes 30-day lookback (`date >= date(?, '-30 days')`); engine's `buildForwardFilledMap()` carries the last known rate forward across weekends/holidays. ECB publishes on business days only — without forward-fill, weekend dates would return null. |
| **D5** | **API cost amortization / caching** | `fx-fetcher.service.ts` — ECB XML feed is ~6MB of historical data, fetched with 30s timeout. Yahoo Finance (`yahoo-finance2`) is a rate-limited fallback. Storing rates in SQLite means one fetch populates years of history; subsequent lookups are pure local queries with no network I/O. |
| **D6** | **Decimal precision** | `extensions.ts` — `rate TEXT NOT NULL`. Stored as text, not `REAL`, to preserve arbitrary-precision Decimal.js values. Matches the codebase convention (all financial values use Decimal, never native float). |
| **D7** | **Auto-detection of required pairs** | `fx-fetcher.service.ts` — `fetchAllExchangeRates()` queries `SELECT DISTINCT currency FROM security UNION SELECT DISTINCT currency FROM account` and fetches all foreign currencies vs `portfolio.currency` (base). Pairs are determined by actual portfolio composition, not hardcoded. |

## 3. Considered Options

### Option A: Mutate the native ppxml2db `price` table directly

**Approach**: Write live-fetched FX rates as rows in `price`, treating exchange rate pairs as pseudo-securities.

**Why rejected:**

- The import pipeline creates a NEW temp DB, populates it exclusively from XML via `ppxml2db.py`, then atomically swaps it in. Every row in `price` comes from the XML; any injected FX rows are destroyed.
- `ppxml2db.py` reads securities/prices from XML nodes; it has no awareness of quovibe-injected data.
- Would require a post-swap migration step to re-inject rates — fragile and contradicts the design principle that ppxml2db tables reflect the XML state.
- `price` uses `BIGINT` (integer × 10^8) for values — FX rates stored as text-based arbitrary-precision decimals don't fit the encoding convention cleanly.

### Option B: Fetch exchange rates on-the-fly (no DB cache)

**Approach**: Call ECB/Yahoo on every performance calculation that needs an FX rate.

**Why rejected:**

- ECB XML feed has a 30-second timeout and returns the entire historical dataset (~6MB XML). Parsing with Cheerio is non-trivial.
- `buildRateMap()` builds dense date-range maps needed by performance calculations (TTWROR, IRR, Statement of Assets). A performance endpoint may need rate maps for 5+ currency pairs across 1-10 years. Fetching on every request is prohibitive.
- Yahoo Finance has rate limits and occasional downtime.
- Forward-fill requires a sorted date sequence in memory; re-parsing XML on every call wastes CPU.

### Option C: JSON sidecar file (like `quovibe.settings.json`)

**Approach**: Store FX rates in a JSON file alongside the DB, avoiding the DB-swap problem entirely.

**Why rejected:**

- FX rates are ~6,000 daily rows per currency pair (ECB back to 1999). A 5-currency portfolio = ~30,000 rows → multi-MB JSON.
- Requires parsing the entire file into memory on every boot (and holding it there), vs. lazy indexed SQLite queries.
- Requires rewriting the entire file on every fetch; SQLite's `INSERT OR REPLACE` touches only the affected rows.
- Application code must implement its own query logic (sorted Map, binary search, 30-day lookback for forward-fill) instead of a single indexed `SELECT`.
- Settings sidecar works because settings are small and unstructured. FX rates are time-series data with range queries — exactly what SQLite is built for.

### Option D (chosen): Custom `vf_exchange_rate` sidecar table

**Approach**: A quovibe-owned table, populated by an explicit fetch action, queried locally by all services that need FX rates. Structurally decoupled from ppxml2db's 24-table namespace.

## 4. The Decision

`vf_exchange_rate` is a **write-once read-many local cache** of historical exchange rates, populated on demand and consumed by the performance calculation layer.

**Architecture role**: A **live-data sidecar** — structurally independent from the PP-imported data, but co-located in the same SQLite file for query simplicity (no second DB connection).

**Data flow**:

```
[Per-portfolio FX scheduler — fires at next 17:00 Europe/Berlin or +6h cap, whichever sooner]
[OR: lazy-fill on GET /api/prices/exchange-rates cache miss]
       │
       ▼
fetchAllExchangeRates()              ← auto-detects currencies from security + account tables
fetchSinglePairOnDemand()            ← lazy-fill path on GET miss
       │
       ├── fetchFromEcb()            ← primary source (ECB XML, EUR-based rates)
       │        or
       └── fetchFromYahoo()          ← fallback (yahoo-finance2, ticker: {FROM}{TO}=X)
              │
              ▼
        saveRates()                  ← INSERT OR REPLACE into vf_exchange_rate
              │
              ▼
 [Performance calculations consume via:]
   fx.service.ts → getRate()         ← single-date forward-fill lookup
   fx.service.ts → buildRateMap()    ← dense date-range map (direct / inverse / EUR triangulation)
              │
              ▼
 [Consumers:]
   performance.service.ts            ← TTWROR, IRR, Statement of Assets
   benchmark.service.ts              ← benchmark index price conversion
   accounts.service.ts               ← deposit account balance conversion
```

The scheduler lives in `packages/api/src/services/fx-scheduler.service.ts`. It
hooks into the `portfolio-db-pool` lifecycle: `setOnOpened` arms a per-portfolio
timer; `setOnEvicted` clears it. On the FIRST open of a real portfolio per
process, the scheduler also eager-fires `fetchAllExchangeRates` once before
arming the cadence tick — mirrors PP's `StartupAddon` ordering
(`provider.update` runs before `Job.schedule(delay)`). A process-scope dedup
Set ensures pool eviction + reopen does NOT re-run the eager fetch; the
scheduled tick handles freshness from then on. Demo portfolios are skipped
entirely (their seeded random walk would be visibly clobbered by live ECB
rates at the seam). There is no manual "Update FX" button — ECB is free,
unlimited, and the 17:00 anchor matches its publish window.

**Key characteristics**:

| Aspect | Design |
|--------|--------|
| Schema | `(date, from_currency, to_currency)` PK → `rate TEXT` |
| Convention | Multiply: `foreignAmount × rate = baseAmount` |
| Precision | Decimal.js via TEXT column (no floating-point) |
| Freshness | Auto on import/startup; manual refresh via Settings |
| Triangulation | Automatic EUR pivot for non-EUR/non-direct pairs |
| Forward-fill | Last known rate carried forward (no interpolation) |
| Upsert | `INSERT OR REPLACE` — re-fetching is idempotent |
| DDL source | `extensions.ts` exports `VF_EXCHANGE_RATE_DDL` (single source of truth) |

## 5. Consequences

### Positive

- **Clean data boundary**: quovibe's live FX data is conceptually (and nominally, via `vf_` prefix) separated from PP's imported data. The 24 ppxml2db tables represent the user's portfolio state as of the last XML export; `vf_exchange_rate` represents live market data fetched independently.
- **Purpose-built query model**: Composite PK `(date, from, to)` makes date-range and single-date lookups trivial — a single `SELECT ... WHERE ... ORDER BY date DESC LIMIT 1` with no JOINs.
- **Triangulation support**: The EUR pivot in `buildRateMap()` means any two currencies are reachable without explicit pair configuration. ECB covers ~30 currencies vs EUR; that yields ~900 cross-pairs.
- **Fetch-once, query-forever**: One ECB fetch populates ~6,000+ daily rates per pair (going back to 1999). All subsequent performance calculations are pure local SQLite queries.
- **Consistent precision**: TEXT storage + Decimal.js throughout the pipeline — zero floating-point contamination.

### Negative / Risks

| Issue | Severity | Status | Detail |
|-------|----------|--------|--------|
| Data loss on re-import | CRITICAL | **RESOLVED** | `preserveCustomTables()` in `import.service.ts` copies all `vf_exchange_rate` rows from the live DB (opened readonly) into the new temp DB after `applyExtensions()` creates the empty table. Uses `INSERT OR IGNORE` to avoid conflicts. |
| Empty on first boot | HIGH | **RESOLVED** | `needsFxFetch()` in `fx-fetcher.service.ts` checks if the table is empty AND the portfolio has foreign currencies. Called in `index.ts` after `buildFullApp()` — fire-and-forget so server boot is not delayed. |
| Manual trigger only | MEDIUM | **RESOLVED** | `fetchAllExchangeRates()` is now auto-triggered in two places: (1) after successful XML import in `routes/import.ts` (fire-and-forget via separate DB connection), (2) on startup when `needsFxFetch()` returns true. Manual refresh via Settings remains available. |
| Duplicated DDL | LOW | **RESOLVED** | `VF_EXCHANGE_RATE_DDL` exported from `extensions.ts` as a string constant (co-located with the Drizzle schema). `applyExtensions()` in `verify.ts` imports and executes it. Single source of truth. |
| No pair-level staleness tracking | LOW | Open | The table has no `fetched_at` column. There's no way to know if the EUR/USD data is from yesterday or two years ago without querying `MAX(date)` per pair. |

## 6. Architect's Verdict

**The architecture is sound. The UX issues have been remediated.**

The core decision — a sidecar table decoupled from the PP import lifecycle, with ECB/Yahoo fetching, EUR triangulation, and forward-fill — is well-engineered and appropriate for the problem. The alternative approaches (mutating ppxml2db tables, on-the-fly fetching, JSON sidecar) have clear and valid reasons for rejection.

The four remediation items identified in the original review are now implemented:

1. **Re-import data preservation** — `preserveCustomTables()` opens the live DB readonly during import and copies all FX rate rows into the new DB. Edge cases handled: first import (no live DB), missing table, empty table.
2. **Auto-population after import** — Fire-and-forget `fetchAllExchangeRates()` call in the import route handler after successful reload. Opens its own DB connection since the import handler's original connection is closed.
3. **Startup check** — `needsFxFetch()` detects empty FX table + foreign currencies on boot. Triggers background fetch without delaying server startup.
4. **DDL deduplication** — `VF_EXCHANGE_RATE_DDL` constant in `extensions.ts` is the single source of truth, imported by `verify.ts`.

The only remaining open item is pair-level staleness tracking (LOW severity), which can be addressed if/when a UX for "last updated" is needed.

ADR: Introduction of custom vf_exchange_rate table

  1. Context & Problem Statement

  Quovibe consumes a SQLite database generated by ppxml2db, which converts a Portfolio Performance XML export into 24 relational tables. This database is treated as  
  the imported dataset — a complete snapshot of the user's portfolio as of the last XML export.

  The problem: quovibe needs historical exchange rate lookups for multi-currency performance calculations, but the native ppxml2db schema does not provide a clean,   
  general-purpose FX rate table. PP stores exchange rate data in two places, neither of which is suitable:

  (a) Transaction-level FX: xact_unit.exchangeRate

  Each transaction unit stores the exchange rate that was applied to that specific transaction (packages/api/vendor/xact_unit.sql:8-13):

  exchangeRate VARCHAR(16)  -- arbitrary-precision, per-transaction

  This is per-transaction, not reusable. You cannot ask "what was EUR/USD on 2024-03-15?" — only "what rate was used in transaction XYZ's FOREX unit." Performance    
  calculations need dense date-range lookups, not per-transaction spot values.

  (b) Exchange rate securities: security + price

  PP can store exchange rates as special "securities" with targetCurrency set (packages/api/vendor/security.sql:8), with historical rates in the price table. However:

  - Not guaranteed to exist — these are user-configured in PP; a minimal export may not include them.
  - Query ergonomics are poor — answering "EUR→USD on date X" requires finding the right security (WHERE currency = 'USD' AND targetCurrency = 'EUR'), then joining to
   price, dealing with direction/convention ambiguity.
  - No triangulation — if the user has EUR→USD and EUR→GBP but not USD→GBP, there's no built-in cross-rate.

  Overwrite-survival hypothesis: CONFIRMED

  The critical driver — visible in the code at packages/api/src/services/import.service.ts:113-212:

  1. runImport() creates a fresh temporary database from scratch
  2. ppxml2db.py populates it entirely from the XML file
  3. applyExtensions(newDb) creates an empty vf_exchange_rate via CREATE TABLE IF NOT EXISTS
  4. reloadApp() performs an atomic file swap (index.ts:68-74): the temp DB replaces the live DB

  If quovibe had written live-fetched FX rates into the native price table, every re-import would obliterate them. The developer created vf_exchange_rate as a sidecar
   table to decouple live FX data from the PP import lifecycle.

  The naming convention confirms intent — the vf_ prefix (seen in the extensions.ts:3 comment: "Tabelle proprie di quovibe (non ppxml2db)") explicitly marks the table
   as quovibe-owned, outside the ppxml2db namespace.

  However, there is a critical bug: the current atomic-swap implementation also wipes vf_exchange_rate DATA. The table structure survives (recreated by
  applyExtensions), but the rows do not — there is zero migration logic to copy rates from the old DB to the new one. The sidecar design protects against ppxml2db    
  overwriting the data, but the file-swap strategy defeats it anyway.

  3. Considered Options

  Option A: Mutate the native ppxml2db price table directly

  Approach: Write live-fetched FX rates as rows in price, treating exchange rate pairs as pseudo-securities.

  Why rejected (evidence):

  - import.service.ts:115-198 — The import pipeline creates a NEW temp DB, populates it exclusively from XML via ppxml2db.py, then atomically swaps it in. Every row  
  in price comes from the XML; any injected FX rows are destroyed.
  - ppxml2db.py:133-140 — The converter reads securities/prices from XML nodes; it has no awareness of quovibe-injected data.
  - Would require a post-swap migration step to re-inject rates — fragile and contradicts the design principle that ppxml2db tables reflect the XML state.
  - price uses BIGINT (integer × 10^8) for values — FX rates stored as text-based arbitrary-precision decimals don't fit the encoding convention cleanly.

  Option B: Fetch exchange rates on-the-fly (no DB cache)

  Approach: Call ECB/Yahoo on every performance calculation that needs an FX rate.

  Why rejected (evidence):

  - fx-fetcher.service.ts:24 — ECB XML feed has a 30-second timeout and returns the entire historical dataset (~6MB XML). Parsing with Cheerio is non-trivial.        
  - fx.service.ts:54-98 — buildRateMap() builds dense date-range maps needed by performance calculations (TTWROR, IRR, Statement of Assets). A performance endpoint   
  may need rate maps for 5+ currency pairs across 1-10 years. Fetching on every request is prohibitive.
  - Yahoo Finance has rate limits and occasional downtime (fx-fetcher.service.ts:86-89 — catch block logs failures and returns empty).
  - Forward-fill requires a sorted date sequence in memory; re-parsing XML on every call wastes CPU.

  Option C (chosen): Custom vf_exchange_rate sidecar table

  Approach: A quovibe-owned table, populated by an explicit fetch action, queried locally by all services that need FX rates. Structurally decoupled from ppxml2db's  
  24-table namespace.

  4. The Decision

  vf_exchange_rate is a write-once read-many local cache of historical exchange rates, populated on user demand and consumed by the performance calculation layer.    

  Architecture role: It is a live-data sidecar — structurally independent from the PP-imported data, but co-located in the same SQLite file for query simplicity (no  
  second DB connection).

  Data flow:

  [User clicks "Update Exchange Rates" in Settings]
         │
         ▼
  POST /api/prices/fetch-exchange-rates
         │
         ▼
  fetchAllExchangeRates()              ← auto-detects currencies from security + account tables
         │
         ├── fetchFromEcb()            ← primary source (ECB XML, EUR-based rates)
         │        or
         └── fetchFromYahoo()          ← fallback (yahoo-finance2, ticker: {FROM}{TO}=X)
                │
                ▼
          saveRates()                  ← INSERT OR REPLACE into vf_exchange_rate
                │
                ▼
   [Performance calculations consume via:]
     fx.service.ts → getRate()         ← single-date forward-fill lookup
     fx.service.ts → buildRateMap()    ← dense date-range map (direct / inverse / EUR triangulation)
                │
                ▼
   [Consumers:]
     performance.service.ts            ← TTWROR, IRR, Statement of Assets
     benchmark.service.ts              ← benchmark index price conversion
     accounts.service.ts               ← deposit account balance conversion

  5. Consequences

  Positive

  - Clean data boundary: quovibe's live FX data is conceptually (and nominally, via vf_ prefix) separated from PP's imported data. The 24 ppxml2db tables represent   
  the user's portfolio state as of the last XML export; vf_exchange_rate represents live market data fetched independently.
  - Purpose-built query model: Composite PK (date, from, to) makes date-range and single-date lookups trivial — a single SELECT ... WHERE ... ORDER BY date DESC LIMIT
   1 with no JOINs.
  - Triangulation support: The EUR pivot in buildRateMap() means any two currencies are reachable without explicit pair configuration. ECB covers ~30 currencies vs   
  EUR; that yields ~900 cross-pairs.
  - Fetch-once, query-forever: One ECB fetch populates ~6,000+ daily rates per pair (going back to 1999). All subsequent performance calculations are pure local      
  SQLite queries.
  - Consistent precision: TEXT storage + Decimal.js throughout the pipeline — zero floating-point contamination.

  Negative / Risks

  6. Architect's Verdict

  The architecture is sound. The UX is broken.

  The core decision — a sidecar table decoupled from the PP import lifecycle, with ECB/Yahoo fetching, EUR triangulation, and forward-fill — is well-engineered and   
  appropriate for the problem. The alternative approaches (mutating ppxml2db tables, or on-the-fly fetching) have clear and valid reasons for rejection. The vf_      
  naming convention, the TEXT-based precision, the three-tier lookup strategy, and the auto-detection of required pairs all demonstrate thoughtful design.

  What needs fixing (in priority order):

  1. Re-import data preservation — The reloadApp() file-swap must migrate vf_exchange_rate rows from the old DB to the new DB before the swap. This could be as simple
   as attaching the old DB and running INSERT OR IGNORE INTO vf_exchange_rate SELECT * FROM old.vf_exchange_rate after applyExtensions() creates the empty table in   
  the temp DB.
  2. Auto-population after import — runImport() (or the post-swap hook in reloadApp()) should trigger fetchAllExchangeRates() automatically so multi-currency
  portfolios work immediately.
  3. Startup check — On boot, if vf_exchange_rate is empty and the portfolio has foreign currencies, auto-trigger a fetch (or at minimum, surface a warning in the    
  UI).
  4. Deduplicate DDL — Remove the raw SQL from verify.ts:81-88; let the Drizzle definition in extensions.ts be the single source of truth (or vice versa — pick one   
  canonical location).

  Do NOT rip it out. The PP security + price approach is unsuitable for the reasons enumerated in Sections 1 and 3A. The sidecar pattern is correct; it just needs the
   plumbing to make it invisible to the user.

❯ what if we move the vf table in a json sidecar like the settings json?

● Interesting idea — it would certainly dodge the re-import wipe. But the data shapes are very different:

  Settings JSON: ~50 key-value pairs, <5 KB, read once at boot.

  FX rates: ~6,000 daily rows per currency pair (ECB goes back to 1999). A 5-currency portfolio means ~30,000 rows. That's a multi-MB JSON file that you'd need to:   

  - Parse entirely into memory on every boot (and hold there), vs. today's lazy indexed queries
  - Rewrite entirely on every fetch (INSERT OR REPLACE on one pair currently touches only that pair's rows; JSON means serialize everything)
  - Implement your own query logic — today fx.service.ts:28-32 does WHERE date <= ? ORDER BY date DESC LIMIT 1 as a single indexed lookup. In JSON you'd need to build
   a sorted Map, do binary search, handle the 30-day lookback for forward-fill seeding — all in application code instead of SQLite

  The settings sidecar works because settings are small and unstructured. FX rates are a time-series with range queries — exactly what SQLite is built for.

  The simpler fix: preserve the vf_exchange_rate rows during the DB swap. In reloadApp(), after copying the temp DB and before closing the old handle:

  // Attach old DB to new, migrate quovibe-owned tables
  const newDb = new Database(swapPath);
  newDb.exec(`ATTACH DATABASE '${DB_PATH}' AS old`);
  newDb.exec(`INSERT OR IGNORE INTO vf_exchange_rate SELECT * FROM old.vf_exchange_rate`);
  newDb.exec(`DETACH old`);
  newDb.close();

  Three lines of SQL, zero architecture change. The sidecar design stays intact, the data survives the swap, and all the existing query/triangulation/forward-fill    
  code remains untouched.
